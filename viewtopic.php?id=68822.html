Packages are yes, essentially tar files of precompiled binaries that extend your system.  Go to the 'software' page and update the package list (the local copy of the package list is held only on a RAM disk, so it is lost every time you reboot), then install luci-app-sqm. 

Now you will have another option under the "network" tab, SQM QoS.  Go to that page and configure your SQM.  Interface would be the one connected to your cable modem, and for a starting point, set the speeds to somewhat less than your typical speedtest results.  This should prevent the buffers from accumulating packets faster than the cable system can unload them.  In order for this particular scheme to work, the SQM needs to know the cable speed ahead of time.

mk24
22 Dec 2016, 19:33

A bridge is just a bit of code in the kernel that takes over several network interfaces and allows them to communicate directly with each other.  It also creates a virtual interface where the OS can reach any of them.  As I understand it, placing interfaces into a bridge is like plugging cables into an unmanaged Ethernet switch.  The switch has a memory of what devices (MAC addresses) have been seen on each port.  When a packet comes in on an interface, the destination MAC is examined and the packet is dispatched to the appropriate other interface.  This is almost instant if the system is not busy.  There is nothing to configure, other than turning on and off the STP protocol which detects if a loop has been formed, to prevent packets from needlessly and destructively racing around it.

The interfaces themselves still have their buffers, but the buffer will stay empty if nothing else is happening.

If I were you I'd use a Linux PC with Wireshark to investigate exactly what the app does.  Is it really only on the LAN or is it trying to go to the Internet?  I have a setup that can answer that easily.  It is simply a PC connected to a router and on to the Internet via Ethernet, and its wifi adapter is in AP mode.  These are joined with a bridge in the kernel just like OpenWrt uses.  On later versions of Ubuntu the "network manager" GUI can set all this up.

By connecting a closed-source device to that AP, Wireshark can see exactly what enters and leaves it, and when, and where it is going.

And again, there can be a lot of latency in a phone due to power-saving in the wifi radio.  It would use a lot of the battery to keep the radio on 100% of the time ready to instantly receive a packet.
