Some OEM firmware recovery routine (without USB <==> TTL cable and router dissassembling) was described here:

Create a McDebian WRT3200ACM V1 rootfs USB Key\Drive - [url=https://github.com/Chadster766/McDebian/wiki/3.-Make-Root-File-System]https://github.com/Chadster766/McDebian/wiki/3.-Make-Root-File-System[/url]
Copy FW_WRT3200ACM_1.0.5.176416_prod.img to the USB Key\Drive 
Enable SSH to Telnet in DD-WRT 
Enable Core USB Support, USB Storage and Automount in DD-WRT 
Plug the McDebian rootfs USB Key into the router and reboot 
SSH or Telnet into DD-WRT 
Run the "mount" commad to see where the USB Drive was automounted (Ex: /tmp/mnt/sda1) 
Run the below commands one at a time substituting if needed the path for the USB key 



Code: 

mount -t proc none /tmp/mnt/sda1/proc 
mount -o bind /dev /tmp/mnt/sda1/dev 

chroot /tmp/mnt/sda1 

flash_erase /dev/mtd5 0 0 
flash_erase /dev/mtd6 0 0 
flash_erase /dev/mtd7 0 0 
flash_erase /dev/mtd8 0 0 

nandwrite -p /dev/mtd5 FW_WRT3200ACM_1.0.5.176416_prod.img 
nandwrite -p /dev/mtd7 FW_WRT3200ACM_1.0.5.176416_prod.img

There is also other "back to stock" procedure which does not require USB to TTL cable.

In short words:

Booting back to stock from other patition 
When you install DD-WRT over stock, it installs on a different partition. You can use the following command by entering it in DDWRT commands on the administration tab, Telnet or SSH to determine what partition DD-WRT is running on. 
Code:
ubootenv get boot_part 

The last number it returns is the partition (it will return 1 or 2). In my case, DD-WRT is on partition 1, so to return to Stock, enter (If yours returns 2, put 1): 
Code:
ubootenv set boot_part 2

and then reboot. You can use the command 
Code:
reboot

It means that every time We install firmware it will be installed on non active partition. So every time before flashing we will active stock firmware partition and flash with FACTORY-to-costume_firmware We will have OEM backup firmware. Do I think correctly?

hnyman

You are thinking correctly. If you first switch from Openwrt/LEDE partition to the OEM firmware and then flash from there, the OEM parition remain intact and you always replace the Openwrt/LEDE image.

But you might look more carefully at the command for the partition switching. You might need to set also another uboot env variable "bootcmd" in addition to "boot_part". See discussion and examples at:
https://forum.openwrt.org/viewtopic.php … 92#p353492

Ps. I have been compiling LEDE for wrt3200acm. No special problems.

silelis wrote:

Which distribution do You use mostly?

I compile mostly from LEDE because it is one year ahead in the core system development. Openwrt development pretty much stopped in May 2016 when practically all active core developers left for LEDE (those who had been developing Openwrt-specific features like netifd, procd, base system etc.)

Currently it seems that if there is a merge of Openwrt and LEDE, the work will continue from LEDE codebase, so the Openwrt codebase is pretty much abandoned. Yes, there are some device-specifc PRs being made in Openwrt but seems that nobody merges them in any more. Pretending that Openwrt is alive and well does not help anybody.

With current LEDE master, kernel 4.9 is the default for WRT3200ACM and pretty much everything works.
(well, CPU frequency scaling is still absent, as well as wifi LEDs, but that concerns all builds, I think.)

silelis wrote:

From my point of view the most importent are:
- stable wifi
- openVPN server
- usb /overlay
- dyndns

Wifi driver is the same in all trunk/master builds, so there is no difference. And all other items are just packages that you can include in your own build.

In general, the answer to the question of the best firmware is always "your own". You can include just the packages that you want and can tweak the defaults to your liking etc.
